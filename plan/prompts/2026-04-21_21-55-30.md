# Increment 3 Prompt

Task: Implement Increment 3 for this slot machine project.

Objective:
Write and execute missing unit tests for `src/state/Wallet.js` to verify correct bankroll management behavior, including default balance handling, bet deduction behavior, win additions, and persistence through mocked `localStorage`.

Requirements:
1. Create or complete a Jest test file for `src/state/Wallet.js`.
   - Use the existing project test setup and conventions.
   - Place the tests in `tests/Wallet.test.js` unless the project already uses a different established test filename/location for this module.

2. Cover the following behaviors at minimum:

   A. `getBalance` default behavior
   - Verifies the wallet returns the correct default balance when no prior balance exists in `localStorage`.
   - Verifies initialization behavior is correct when storage is empty or missing the expected key.

   B. `deductBet` success case
   - Verifies a valid positive bet less than or equal to current balance is deducted correctly.
   - Verifies the resulting balance is updated correctly in memory and persisted to `localStorage` if applicable.

   C. `deductBet` insufficient funds case
   - Verifies a bet larger than the current balance is rejected correctly.
   - Verifies balance does not change.
   - Verifies persistence state does not become inconsistent.

   D. `deductBet` zero bet case
   - Verifies behavior when bet amount is `0`.
   - Assert the exact intended behavior based on the current Wallet contract:
     - either reject / throw / return false / no-op, whichever matches the module design.
   - Ensure balance remains correct and persistence remains correct.

   E. `deductBet` negative bet case
   - Verifies behavior when bet amount is negative.
   - Assert the exact intended behavior based on the current Wallet contract.
   - Ensure invalid values do not increase balance or corrupt state.

   F. `addWin` positive case
   - Verifies a positive win amount increases balance correctly.
   - Verifies updated balance is persisted correctly.

   G. `addWin` zero case
   - Verifies behavior when win amount is `0`.
   - Assert the exact intended behavior based on the current Wallet contract.
   - Ensure no incorrect state change occurs.

   H. `localStorage` persistence
   - Mock `localStorage` in tests.
   - Verify reads and writes happen as expected.
   - Verify a Wallet instance can restore previously saved balance from mocked storage if that is part of the module behavior.

3. Use mocked `localStorage` rather than real browser storage.
   - Tests must be deterministic and isolated.
   - Clear mocks and reset storage state between tests.

4. Keep the work limited to Increment 3 scope only.
   - Do not change game logic.
   - Do not refactor unrelated files.
   - Do not perform Increment 4 controller/view refactor.
   - Do not add UI features.
   - Do not modify payout logic.
   - Only make the smallest production-code change if tests reveal Wallet is missing minor support needed for its intended contract.

5. Prefer testing the current public API of `Wallet.js`.
   - Do not test internal implementation details unnecessarily.
   - Focus on externally observable behavior.

6. After writing the tests, run the Wallet test file and confirm whether it passes.
   - If project conventions support it, also run the full test suite to ensure no regressions.

Expected deliverables:
1. A Jest test file covering the Wallet behaviors listed above.
2. Mocked `localStorage` setup used by the Wallet tests.
3. A short summary stating:
   - which test cases were added
   - whether any production code had to be adjusted
   - which files were modified
   - whether the Wallet tests pass
   - whether the full test suite passes, if run

Constraints:
- Keep changes minimal and maintainable.
- Do not alter runtime behavior beyond enforcing the intended Wallet contract.
- If the current Wallet implementation’s contract is ambiguous, infer it from the existing code and test the most reasonable intended behavior consistently.
- If a behavior is clearly buggy, fix it only as much as needed to support correct bankroll management and passing tests.

Success condition:
`src/state/Wallet.js` has robust Jest coverage for default balance, valid/invalid bet deduction, win addition, and mocked `localStorage` persistence, and the tests execute successfully.
