# Increment 16 Prompt

**Task:** Implement a basic daily login/retention mechanic for this slot machine project.

**Objective:**  
Add a simple retention system that encourages daily play by granting rewards based on login frequency or balance status (e.g., a flat bonus if the balance is zero, or tracking a daily streak with incremental rewards). Store data persistently (e.g., using localStorage) to track user engagement across sessions. Integrate the mechanic into the GameManager and UI without disrupting core gameplay.

**Requirements:**

1. **Data Persistence and State Management:**
   - Implement storage for daily login data (e.g., last login date, streak count) using localStorage or a similar client-side mechanism.
   - Track daily streaks: Increment streak on consecutive logins; reset on missed days.
   - Define reward logic: Grant a flat bonus (e.g., $10) if balance is zero on login, or scale rewards based on streak (e.g., +1 bonus per streak day, up to a cap).
   - Ensure data is loaded on app initialization and updated after rewards are applied.

2. **UI Elements and Feedback:**
   - Add a display in the UI to show current streak, last login date, and any pending rewards.
   - Include a notification or popup on login to inform users of earned bonuses.
   - Place elements in a non-intrusive area, such as the top bar or a dedicated section.

3. **GameManager Integration:**
   - Modify GameManager to check and apply daily rewards on initialization or first interaction.
   - Ensure rewards are added to the Wallet only once per day/session.
   - Prevent abuse (e.g., no rewards on same-day reloads) by validating timestamps.
   - Update UI after applying rewards to reflect balance changes.

4. **Reward Application Logic:**
   - Apply bonuses conditionally: Only if balance is zero for the flat bonus, or based on streak length.
   - Cap rewards to prevent exploitation (e.g., max streak bonus of $50).
   - Log or display reward history for transparency.

5. **Testing and Coverage:**
   - Write unit tests for GameManager daily logic, mocking storage and Wallet.
   - Cover scenarios: First login, streak continuation, reset, reward application, and edge cases (e.g., date manipulation).
   - Run full lint and test suite to ensure no regressions.

**Expected deliverables:**

1. Updated `src/controller/GameManager.js` with daily reward logic and persistence handling.
2. Modified `src/ui/view.js` with UI elements and updates for streak/reward display.
3. Modified `src/state/Wallet.js` if needed for bonus application.
4. Unit tests in `tests/GameManager.test.js` for daily mechanic behaviors with mocked dependencies.
5. Confirmation that lint passes.
6. Confirmation that the full test suite passes.
7. A short summary stating:
   - Which daily behaviors were implemented and tested.
   - Which dependencies were mocked.
   - Whether any production code was adjusted.
   - Which files were modified.
   - Whether lint passes.
   - Whether all tests pass.

**Constraints:**

- Keep changes minimal and focused on the retention mechanic; do not alter core spin or payout logic.
- Preserve existing runtime behavior for gameplay.
- Use client-side storage only; avoid server-side dependencies.
- Enforce responsible gaming: Rewards should encourage engagement without promoting addiction.
- Do not add complex features like server sync or advanced analytics.
- If ambiguities arise, infer reasonable behavior from existing code.

**Success condition:**  
The daily login/retention mechanic is implemented with streak tracking and conditional bonuses, integrated into GameManager and UI, and includes unit tests for key behaviors using mocked dependencies. The project passes full lint and test checks with no regressions, and users receive appropriate rewards on daily engagement.
