# Increment 12 Prompt

**Task:** Implement the Auto-Spin feature for this slot machine project.

**Objective:**  
Add an auto-spin mode to the UI that allows automated spinning with mandatory stop conditions to prevent unconditional autoplay. Wire the feature to the GameManager for state management, validation, and enforcement of stop criteria. Ensure the feature integrates seamlessly with existing spin logic and provides user control to start/stop at any time.

**Requirements:**

1. **UI Elements:**
   - Add an auto-spin toggle button (e.g., "Auto-Spin" that switches to "Stop Auto-Spin" when active) in the controls area.
   - Include input fields or dropdowns for setting stop conditions (e.g., number of spins, balance threshold, win/loss amount).
   - Add a status display to show auto-spin progress (e.g., spins remaining, current condition status).
   - Ensure elements are accessible and provide clear labels for responsible gaming.

2. **UI Logic:**
   - Initialize references to auto-spin button, condition inputs, and status display.
   - Bind events to start auto-spin only if valid stop conditions are configured; prevent activation without them.
   - Handle stopping auto-spin immediately on user command, updating the UI to reflect the change.
   - Update the status display in real-time during auto-spin (e.g., after each spin), and disable manual spin button during auto-mode.

3. **GameManager Logic:**
   - Add state variables for auto-spin mode (e.g., active flag, configured conditions).
   - Implement functions for starting and executing auto-spin:
     - Validate stop conditions before starting (no unconditional autoplay).
     - Perform spins sequentially, checking conditions after each (e.g., balance check, spin count).
     - Stop automatically when any condition is met or on manual interrupt.
     - Reuse existing spin logic without duplication, ensuring proper sequencing of bet deduction, math evaluation, and UI updates.
   - Ensure auto-spin respects game guards (e.g., no overlapping spins) and resets state correctly.

4. **Integration and Validation:**
   - Integrate with Wallet for balance checks and updates during auto-spin.
   - Ensure stop conditions are enforced strictly (e.g., halt on insufficient funds, big win/loss).
   - Allow manual interruption at any point, with immediate feedback.
   - Test for edge cases: rapid condition changes, invalid inputs, and integration with manual spins.
   - Include user warnings or confirmations for high-stakes settings to promote responsible play.

5. **Testing and Coverage:**
   - Write unit tests for GameManager auto-spin behavior, mocking dependencies like Wallet and math module.
   - Cover key paths: starting with valid conditions, stopping on conditions or manually, sequencing of spins, and state resets.
   - Run full lint and test suite to ensure no regressions.

**Expected deliverables:**

1. Updated `src/index.html` with auto-spin UI elements.
2. Modified `src/ui/view.js` with event bindings and UI updates for auto-spin.
3. Modified `src/controller/GameManager.js` with auto-spin state and logic.
4. Unit tests in `tests/GameManager.test.js` for auto-spin functionality with mocked dependencies.
5. Confirmation that lint passes.
6. Confirmation that the full test suite passes.
7. A short summary stating:
   - Which auto-spin behaviors were implemented and tested.
   - Which dependencies were mocked.
   - Whether any production code was adjusted.
   - Which files were modified.
   - Whether lint passes.
   - Whether all tests pass.

**Constraints:**

- Keep changes minimal and within the feature scope; do not add unrelated features.
- Preserve existing runtime behavior for manual spins.
- Enforce stop conditions strictly; no unconditional autoplay.
- Use mocks/stubs for testing to isolate GameManager.
- Do not modify payout logic, UI beyond auto-spin, or add end-to-end tests.
- If ambiguities arise, infer reasonable behavior from existing code.

**Success condition:**  
The auto-spin feature is implemented with mandatory stop conditions, properly wired to GameManager, and includes unit tests for key behaviors using mocked dependencies. The project passes full lint and test checks with no regressions, and users can start/stop auto-spin safely with enforced limits.
