# Increment 4 Prompt

Task: Implement Increment 4 for this slot machine project.

Objective:
Refactor the currently coupled manager logic in `src/main.js` into a DOM-free controller module and a separate view module, while preserving all existing behavior. After the refactor, `src/main.js` should be only a thin entry point that wires the controller and view together. The spin loop and current UI behavior must continue to work end-to-end with no behavior change.

Required architecture after refactor:

1. `src/controller/GameManager.js`
   - Contains controller / orchestration logic only.
   - Must be DOM-free.
   - Must not call `document.querySelector`, `getElementById`, or directly manipulate HTML elements.
   - Must coordinate game flow, wallet interactions, math logic, spin state, and communication with the UI through injected callbacks, method calls, or event-style interfaces.

2. `src/ui/view.js`
   - Contains DOM querying, DOM updates, UI rendering, button hookups, visual state toggles, and other browser-facing behavior.
   - Must not contain core game decision logic that belongs in the controller.
   - May expose methods the controller can call, and may accept callbacks so user actions can trigger controller methods.

3. `src/main.js`
   - Becomes a thin composition root / entry point only.
   - Instantiates dependencies and wires them together.
   - Should initialize the Wallet, slot math module, controller, and view.
   - Should not continue to contain mixed controller + DOM logic after the refactor.

Refactor requirements:

1. Move the current manager/controller behavior out of `src/main.js` and into `src/controller/GameManager.js`.
2. Move DOM-related behavior out of `src/main.js` and into `src/ui/view.js`.
3. Preserve the existing public behavior of the application.
4. Preserve the current working spin loop end-to-end.
5. Preserve current UI layout and visible interactions unless a very small change is required to support separation of concerns.
6. Keep the current `SlotMachineMath` and `Wallet` integration working correctly.
7. Keep diffs as small as possible while still achieving the architecture goal.
8. Do not introduce new features.
9. Do not alter payout logic.
10. Do not alter game math.
11. Do not add UI polish work.
12. Do not perform Increment 5 tests in this step unless absolutely necessary to keep the project runnable.
13. Do not add unrelated refactors.

Controller expectations:

- The controller should manage game flow such as:
  - preventing duplicate spins while already spinning
  - validating / consuming the bet through Wallet
  - triggering the spin sequence
  - receiving the spin result from the math/game logic
  - calculating or receiving outcomes as already designed
  - instructing the view how to update
  - applying winnings through Wallet
  - signaling spin start / spin end / error or insufficient-funds states
- The controller should interact with the view through an explicit interface rather than touching the DOM directly.

View expectations:

- The view should manage:
  - DOM element lookup
  - display of balance, bet, messages, and reel/grid updates
  - button event listeners and forwarding those actions to the controller
  - visual changes needed during spin state
- The view should not own bankroll logic or payout rules.

Implementation guidance:

- Prefer dependency injection so `GameManager` receives its dependencies explicitly, such as Wallet, SlotMachineMath, and a view interface.
- If helpful, define a small clear interface between controller and view.
- Use the existing code structure and naming conventions where reasonable.
- Favor minimal restructuring over large rewrites.

Behavior preservation:

- No behavior change is allowed except internal code organization needed for separation.
- The spin loop must still function exactly as before from the user perspective.
- Existing controls must continue to work.
- Existing balance updates and win/loss handling must continue to work.

Expected deliverables:

1. New DOM-free controller module at `src/controller/GameManager.js`
2. New UI/DOM module at `src/ui/view.js`
3. Refactored `src/main.js` as a thin entry point
4. Existing app still runs correctly end-to-end
5. A short summary stating:
   - what logic moved into `GameManager.js`
   - what logic moved into `view.js`
   - what remains in `main.js`
   - whether the spin loop still works
   - which files were modified

Constraints:

- Keep the code maintainable and easy to test in the next increment.
- Do not over-engineer.
- Do not add frameworks.
- Do not change runtime behavior.
- Keep the controller isolated from the DOM.

Success condition:
The project is refactored into a 3-layer structure for this portion of the app: `GameManager.js` contains DOM-free controller logic, `view.js` contains DOM/UI logic, `main.js` is only wiring, and the application still works exactly as before, including the full spin loop.
